ТЕХНИЧЕСКОЕ ЗАДАНИЕ

ТрайВайс — это SaaS-платформа, которая позволяет интернет-магазинам внедрить на свои сайты виджет виртуальной примерки одежды с использованием AI.

Покупатель загружает свое фото, выбирает товар, и система за считанные секунды генерирует реалистичное изображение с примеркой выбранной вещи.

Владельцы магазинов получают админ-панель с аналитикой примерок, управлением товарной базой, контролем расходов, баланса и оплаты.

Ценность для бизнеса: Снижение возвратов на 30-40% за счет визуализации посадки вещи, повышение конверсии на 15-25%, сбор ценных данных о предпочтениях покупателей.

Технологический стек:

Бэкенд и база данных:
Supabase (облачная версия)
Фронтенд (админка, PWA): Next.js 14+ (App Router, Server Components, API Routes)
Виджет: Web Component Vanilla JavaScript
AI-генерация: NanoBanana API
Платежи: CloudPayments
Хостинг: Timeweb Cloud App Platform

Цели и задачи проекта:

1. Разработать легковесный асинхронный виджет для встраивания на сайты клиентов.
2. Создать SaaS-админку (PWA) для управления клиентами, товарами и аналитикой.
3. Интегрировать AI-обработку изображений для проверки качества фото перед примеркой.
4. Обеспечить безопасное хранение пользовательских данных и соответствие 152-ФЗ (согласие на обработку).
5. Внедрить биллинговую систему с помесячной оплатой через CloudPayments.
6. Реализовать автоматическую загрузку товаров через XML-фид с анализом изображений и делением на изображения товара и изображения образов.

Пользовательские роли

1. Владелец магазина (Клиент SaaS):
   - Регистрируется в системе через PWA-админку.
   - Настраивает свой магазин: название, адрес URL и подключает XML-фид с товарами, есть также возможность указать что должен содержать URL, чтобы на нем инициализировался виджет. - есть возможность добавить несколько магазинов под одним аккаунтом
   - Получает уникальный `shop_id` и скрипт для вставки на сайт.
   - Управляет товарами загруженными из XML-фида (есть возможность переключаться между магазинами) - включает/отключает появление виджета на том или ином товаре, позволяет увидеть статистику кол-ва примерок с каждым товаром и переход в раздел аналитики, с предустановленным фильтром на этот товар. Товары отображаются в виде таблицы: название товара / цена / категория / доступность / кол-во фото используемых для генерации / ссылка в магазин / статистика + ссылка / вкл/выкл
   - Имеет возможность открыть раздел аналитики, в котором отображается статистика в виде графика (кол-во показов виджета, кол-во открытий виджета, кол-во переходов к генерации, кол-во успешных генераций). Есть фильтр по диапазону, а также возможность указать данные на графике для конкретного товара, или категории товаров (категории передаются в товарном фиде).
   - Оплачивает помесячный тариф с ограничением по количеству примерок.

2. Покупатель (Посетитель сайта магазина):
   - Открывает виджет на странице товара.
   - Загружает свое фото (дает согласие на обработку нажимая кнопку загрузить, снизу под кнопкой подпись “Нажимая на кнопку «Загрузить фото», вы соглашаетесь с Политикой конфиденциальности и Обработкой персональных данных”).
   - Может загрузить так несколько изображений
   - Получает сгенерированное изображение с примеркой - предварительно выбрав что будет примерять, отдельно товар или Образ.
   - Может скачать или поделиться результатом, а также детально посмотреть изображение - нажав на лупу.
   - В течение 30 дней может вернуться к своим генерациям.
   - Может сделать новую генерацию, выбрав другую свою фотографию или другой тип генерации: отдельно товар или Образ.

3. Администратор платформы:
   - Управление всеми клиентами и их тарифами - блокирует в случае необходимости.
   - Мониторинг работы системы.
   - Управление тарифными планами и лимитами.
   - Просмотр общей аналитики по платформе.
   - Задаем промпты для каждого магазина используемые в генерации фотографий для клиентов.
  - в админке также редактирует промпты используемые для: 1 для анализа типа фото товара из фидов Полный образ/Один товар; 2 для анализа загружаемых пользователями фотографий, на предмет подходит или нет для генерации

Функциональные требования

1. Система виджета (Core Feature)

1.1. Установка и инициализация:
- Виджет внедряется на сайт клиента через HTML-скрипт с data-атрибутами:
  ```html
  <script 
    src="https://app.tryvice.ru/widget.js" 
    data-shop-id="SHOP_ID"
    data-product-id="PRODUCT_ID"
    data-button-selector=".fitwise-button"
  ></script>
  ```
  
  Параметры data-атрибутов:
  - `data-shop-id` (обязательный) — уникальный идентификатор магазина в системе
  - `data-product-id` (опциональный) — идентификатор товара на странице, если не задан, виджет будет использовать ссылку страницы для последующего определения что это за товар
  - `data-button-selector` (опциональный) — CSS-селектор кнопки, при клике на которую открывается виджет. Если не указан, виджет создает свою кнопку в нижнем правом углу.

- Скрипт загружается асинхронно (`async/defer`) и не блокирует рендеринг сайта.
- При инициализации виджет проверяет:
  - Соответствие домена сайта разрешенному домену клиента (защита от подмены).
  - Наличие валидного `shop_id`.
  - Доступность API и баланса клиента.
  - Также проверяет что находится на разрешенных клиентом страницах (проверяя настройку есть также возможность указать что должен содержать URL, чтобы на нем инициализировался виджет)
  - Если баланс клиента исчерпан — кнопка скрывается, уведомления не выводятся (тихое выключение).
 - если есть какие либо ошибки, выводятся в консоли

1.2. Пользовательский сценарий виджета:

Шаг 1: Выбор товара
- Виджет автоматически считывает `product_id` из data-атрибута скрипта.
- Если товар доступен в базе клиента, отображается кнопка открытия виджета (или активируется существующая).

Шаг 2: Загрузка фото
- Пользователь загружает свое фото (JPG, PNG, WEBP, максимум 10MB).
- Предварительная ИИ-проверка:
  - Фото проверяется на наличие человека в полный рост (не обрезанное лицо/плечи).
  - Оценивается качество фона (не должно быть слишком много деталей, людей, текстур).
  - Проверяется освещение и контрастность.
- Защита от дублирования: Вычисляется хеш (или сумма) изображения. Если пользователь повторно загружает то же фото, анализ не запускается повторно, а используется результат предыдущей проверки.
- Если фото не подходит → виджет просит загрузить другое с конкретными рекомендациями (например: "Сделайте фото в полный рост на светлом фоне").
- Согласие на обработку: Перед отправкой фото виджет запрашивает разрешение:
под кнопкой текст согласия: “Нажимая на кнопку «Загрузить фото», вы соглашаетесь с Политикой конфиденциальности и Обработкой персональных данных”

Шаг 3: Выбор варианта примерки

- Варианты примерки (фотографии товара берется из товарного фида, который при подключении анализируется ИИ чтобы определить к какому типу относится фото Одна вещь/Полный образ  - тип вещи сохраняется в базе, как и остальные данные фида, если появилось новое изображение (должны определить по хешу или сумме), оно должно пройти анализ заново, анализировать будем через отдельную ИИ модель):
  - Одна вещь: Стандартный режим, примеряется один товар.
  - Полный образ: Если в фиде для товара переданы изображения образа

Шаг 4: Генерация
- Виджет отправляет запрос на API (Next.js API Route):
  ```json
  {
    "shop_id": "uuid",
    "product_id": "123", (или ссылку на которой находится, ссылка товара)
    "image_base64": "data:image/jpeg;base64,...",
    "type": "single" | "outfit"
  }
  ```
- API валидирует данные и отправляет запрос в NanoBanana.
- Время генерации: 5-15 секунд.
- Асинхронный режим: Пока идет генерация, пользователь видит прогресс-бар или анимацию.

Шаг 5: Результат
- Сгенерированное изображение отображается в виджете.
- Доступные действия:
  - Скачать изображение (в формате PNG/JPEG).
  - Посмотреть примерку детально (всплывающее окно на весь экран и лупа).
  - Поделиться ссылкой (по ссылке мы открываем брендированный просмотрщик изображения в канвасе, чтобы его не было так просто скачать - изображение храним в Supabase storage).
  - Закрыть виджет — при этом:
    - Генерация сохраняется в куках пользователя (`fitwise-history`) на 30 дней.
    - При повторном открытии виджета на этом же устройстве пользователь видит свою историю примерок (чтобы вернуться к действиям), а также свои уже загруженные фотографии (которые может использовать для примерок).


1.3. Управление сессией и куки:
- Кука `fitwise-session` устанавливается на 30 дней.
- Хранит:
  - `visitor_id` (анонимный идентификатор).
  - Список `generation_ids` (массив ссылок на сохраненные результаты).
- Список своих фотографий - Последние 5 загруженных изображения (хеши для защиты от повторного анализа).
- Все данные в куках шифруются (AES-256).

2. Бэк-офис (SaaS Админка PWA)

2.1. Аутентификация и мультитенантность:
- Регистрация новых клиентов (владельцев магазинов) через Supabase Auth.
- Поддержка входа по email/password.
- Каждый клиент видит **только свои данные** благодаря Row Level Security (RLS) в PostgreSQL.

2.3. Просмотр статистики и генераций:
В админке клиент видит сводный дашборд генераций (не является разделом настроек или аналитики): 
- дата (по умолчанию сегодняшний день, можно выбрать диапазон по дням/неделям/месяцам)
- список товаров участвующих в генерации и кол-во генераций по ним
- кол-во уникальных пользователей которые сгенерировали
- Количество примерок.
- Расход баланса: количество использованных примерок / из доступных.
- Также в админке есть уже описанные выше разделы Настройки / Аналитики
+ к ним нужен раздел с “Просмотр примерок” в виде строк: Фото примерки (миниатюра и ссылка) / Фото товара (миниатюра и ссылка) / Фото клиента (миниатюра и ссылка) / дата / название товара и ссылка на него / кнопка поделиться примеркой  - все также можно указать период и поставить фильтр на товар (фильтр на товар во всех разделах делаем через поисковую форму, чтобы можно было в ней указать название товара или идентификатор)

2.4. Управление изображениями:
- Все пользовательские фото и сгенерированные изображения хранятся в Supabase Storage.
- Жизненный цикл:
  - Срок хранения: **180 дней** (6 месяцев) с момента последнего доступа.
  - Если к изображению не обращались 180 дней → удаляется автоматически (cron job в Supabase Edge Functions).
  - Запросы к изображениям логируются (таблица `image_access_log`).

2.5. Тарифы и оплата:
- Модель оплаты: Помесячная подписка с фиксированным количеством примерок.
- Тарифные планы (пример):
  | Тариф | Стоимость/мес | Количество примерок |
  |-------|---------------|---------------------|
  | Starter | 5 000 ₽ | 50 примерок |
  | Business | 20 000 ₽ | 200 примерок |
  | Pro | 50 000 ₽ | 500 примерок |
- При оплате через CloudPayments баланс клиента пополняется на соответствующее количество примерок.
- Автоматическое списание: Каждая успешная генерация уменьшает счетчик `remaining_generations`.
- При достижении нулевого баланса:
  - Виджет скрывает кнопку на сайтах клиента (email-напоминание владельцу).
  - В админке отображается уведомление о необходимости пополнения.
  - За 3 дня до окончания тарифа отправляется email-напоминание.

- Использование официального облачного сервиса Supabase с выбором региона "Europe (Frankfurt)" — ближайший к РФ.
- Более быстрое развертывание, меньше DevOps-нагрузки.

Хостинг Next.js приложения:
- Timeweb Cloud App Platform

3.2. База данных Supabase (Основные таблицы):

3.3. API Эндпоинты (Next.js API Routes) предлагаемые

| Метод | Эндпоинт | Описание |
|-------|----------|----------|
| POST | `/api/widget/init` | Инициализация виджета: проверка client_id, домена, баланса |
| POST | `/api/widget/tryon` | Запрос на генерацию примерки |
| GET | `/api/widget/status/:generation_id` | Получение статуса генерации (для асинхронного опроса) |
| GET | `/api/widget/history` | История генераций для конкретного visitor_id |
| POST | `/api/widget/analyze-image` | AI-проверка изображения перед примеркой |
| POST | `/api/admin/products/sync` | Синхронизация товаров из XML-фида |
| GET | `/api/admin/products` | Получение списка товаров клиента |
| GET | `/api/admin/analytics` | Статистика для админки |
| POST | `/api/admin/generations` | Получение списка генераций с фильтрами |
| POST | `/api/webhook/cloudpayments` | Веб-хук для обработки оплат |
| GET | `/api/admin/balance` | Проверка текущего баланса |
| GET | `/api/admin/tariffs` | Получение списка тарифов |

3.4. Алгоритм работы виджета:

```
Покупатель → Виджет → API → NanoBanana → Supabase → Результат
   1. Загрузка страницы товара
   2. Виджет инициализируется с data-атрибутами
   3. Запрос к API: валидация shop_id, домена и баланса
   4. Если баланс > 0 → отображается кнопка, иначе скрывается (без уведомлений)
   5. API возвращает настройки виджета (цвета, текста и пр. настройки - дать возможность владельцу магазина указать в настройках админки какого цвета будет кнопка в виджете, а также какой текст согласия будет под кнопкой, с возможностью установить там ссылку - это должен быть html кусок кода)
   6. Покупатель нажимает на кнопку
   7. Покупатель загружает фото
   8. Виджет отправляет фото на AI-проверку (через API)
   9. API проверяет хеш → если есть в кеше, использует результат
   10. Виджет открывается и показывает выбор режима (одна вещь / образ)
   11. Отправка в NanoBanana
   12. Ожидание генерации (polling или WebSocket через Realtime)
   13. Сохранение результата в Supabase Storage + таблица generations
   14. Списывание 1 примерки из баланса клиента
   15. Отображение результата в виджете
   16. Сохранение ссылки в куках пользователя (30 дней)
```

Нефункциональные требования

1. Производительность:
   - Время загрузки виджета (без учета генерации): < 500ms.
   - Время генерации: не более 20 секунд (среднее 5-10 секунд).
   - Поддержка до 1000 одновременных пользователей на один инстанс.

2. Безопасность:
   - Все запросы между виджетом и API через HTTPS.
   - CORS ограничен разрешенными доменами клиентов.
   - Хранение изображений в приватных бакетах Supabase (доступ только по сгенерированным ссылкам с TTL).
   - Валидация всех входных данных (размер, тип файлов, XSS-атаки).
   - Шифрование кук (AES-256).

3. Надежность:
   - Retry-механизм для запросов к NanoBanana (максимум 3 попытки).
   - Fallback-сообщение при ошибке: "Что-то пошло не так, попробуйте позже".
   - Логирование всех ошибок в таблицу `error_logs`.

4. Масштабируемость:
   - Возможность горизонтального масштабирования Next.js через Timeweb Cloud.
   - Использование индексов в PostgreSQL для ускорения запросов.

5. Юзабилити:
   - Адаптивный дизайн (мобильные, планшеты, десктопы).
   - Поддержка темной и светлой темы (настраивается в админке).
   - Анимация загрузки и прелоадер для улучшения UX.



Администратор платформы: расширенные возможности

Администратор имеет доступ к панели управления платформой со следующими функциями:

1. Управление клиентами:
   - Просмотр всех зарегистрированных клиентов и их магазинов.
   - Редактирование тарифов и количества примерок.
   - Блокировка/разблокировка клиентов.
   - Просмотр истории платежей каждого клиента.

2. Управление тарифами:
   - Создание, редактирование и удаление тарифных планов.
   - Установка цены и количества примерок для каждого тарифа.
   - Настройка специальных условий для Enterprise-клиентов (непубличные тарифы, не доступны при регистрации и оплате)

3. Мониторинг системы:
   - Дашборд с общей статистикой (все клиенты, все генерации).
   - Графики загрузки системы (количество запросов в минуту).
   - Просмотр ошибок и логов.

4. Поддержка клиентов:
   - Отправка уведомлений клиентам.
